# Env Var Exposure Mechanism — Design Rationale

### Panel Deliberation

Five panelists (systems, web/scripting, PLT, DevOps/tooling, AI/ML) voted independently on 3 questions about how env var operations should be exposed to Blink users. Majority consensus on all three questions, with PLT dissenting on Q1 and Q3.

**Context:** The Env effect API surface was already decided (5-0 unanimous, see [env-effect-api.md](env-effect-api.md)). This deliberation addresses the *implementation mechanism*: how `env.var()`, `env.vars()`, `env.cwd()`, `env.set_var()`, `env.remove_var()`, and `env.exit()` get wired from Blink source to C output. Currently `env.args()` works via codegen builtin, `get_env()` exists as an untracked builtin (soundness hole), and the C runtime has `blink_env_vtable` with `read`/`write` function pointers that codegen doesn't use.

**Q1: Implementation mechanism (4-1 for A: Pure codegen builtin with vtable dispatch)**

- **Systems (A):** The vtable pattern is already established and working. `io.println()` emits `__blink_ctx.io->print(...)`, `time.now()` emits `__blink_ctx.time->read()`. Env operations follow identically — `env.var("PORT")` emits `__blink_ctx.env->read("PORT")`. A stdlib `.bl` module would be a pass-through wrapper adding a function call frame for zero benefit. These are leaf calls into libc (`getenv`, `setenv`, `unsetenv`, `getcwd`). The vtable already exists in `runtime_core.h` — we extend it with `cwd`, `args`, `vars`, `remove`, `exit_fn` fields and add default implementations.
- **Web (A):** From the user's perspective, `env.var("PORT")` "just works" when you declare `! Env.Read`. Users don't care whether there's a stdlib module underneath — they care that autocomplete shows `env.var`, `env.cwd`, etc. when they type `env.`. A stdlib module is invisible plumbing that doesn't improve DX. JS devs are used to `process.env.PORT` being a language-level feature, not an imported module.
- **PLT (B):** *(dissent)* Effects must be abstract over their implementation. Algebraic effects decouple interface (effect signature) from handler (implementation). Hardcoding in codegen welds interface to one handler, breaking the algebraic property — handlers can no longer substitute freely. A stdlib module should provide the default handler, bottoming out in C calls. The compiler can optimize the default handler path with inline C (like Koka optimizes known handlers), but this is an optimization, not the semantic model. Cross-language precedent: Koka (`std/os/env`), Eff, and OCaml 5 all model env as a standard effect with a library-provided default handler.
- **DevOps (A):** Codegen builtins mean zero import ceremony. The user writes `env.var("PORT")` and the compiler handles everything. No `import std.env` needed. Error messages come from the compiler directly: `error[E0412]: env.var() requires effect '! Env.Read'`. With a stdlib module, the error would be about a missing import first, then a missing effect — two errors instead of one.
- **AI/ML (A):** One mechanism, one pattern. `env.var()` works exactly like `io.println()` — declare the effect, call the method. LLMs that have seen one effect handle example can generate all others. A stdlib module would add an import step that LLMs might forget, creating a two-step failure mode (import error → effect error). The codegen builtin pattern eliminates the import step entirely.

**Q2: Legacy `get_env()` builtin (4-1 for A: Remove immediately)**

- **Systems (A):** `get_env()` calls raw `getenv()`, bypassing `__blink_ctx.env->read()`. Any function can call it without declaring `! Env.Read`. This is a capability violation — not a deprecation candidate, but a bug. Blink is pre-1.0 with no backwards compatibility contract. Rip it out and replace all call sites with `env.var()`. The compiler's own code using `get_env()` must declare `! Env.Read` or use internal access — this is good dogfooding.
- **Web (B):** Deprecation is gentler. Some users might have scripts using `get_env()`. A warning that says "get_env() is deprecated, use env.var() instead" gives them a migration path. But honestly, Blink is pre-1.0 — maybe removal is fine. *(dissent — weak preference for B)*
- **PLT (A):** `get_env()` is an effect-system soundness hole. A function with no declared effects can observe mutable external state. This violates the fundamental guarantee: if a function's signature says it's pure, it's pure. Every moment `get_env()` exists, the effect system is unsound. There is no principled way to keep it — not as an escape hatch, not as a deprecated feature. Remove it.
- **DevOps (A):** Clean error messages matter. If `get_env()` is removed, users get: `error[E0001]: unknown function 'get_env'. Did you mean 'env.var()'?`. One clear migration path. If deprecated, the warning and the suggestion coexist in a confusing way. Remove + good error message = best migration story.
- **AI/ML (A):** Two ways to read env vars means LLMs generate the wrong one 50% of the time. `get_env()` has training signal from Go (`os.Getenv`), Python (`os.getenv`), etc. An LLM might generate `get_env("PORT")` when it should generate `env.var("PORT")`. Removing the ambiguity eliminates the decision point entirely. One pattern, learnable from 2 examples.

**Q3: Handler mockability (4-1 for A: Vtable swap)**

- **Systems (A):** Already implemented. Codegen emits `void* save = (void*)__blink_ctx.env; __blink_ctx.env = handler_vtable;` for handler blocks. Same pattern as IO, Time, FS. Single pointer dereference for dispatch (~2ns). No reason to introduce a second mechanism.
- **Web (A):** Mock handlers in tests work exactly like the spec shows — `with handler Env { ... } { ... }`. The vtable swap is invisible to users. It "just works." No DX impact from the implementation choice.
- **PLT (B):** *(dissent)* Env must use the same interception mechanism as every other effect — evidence passing. If Env used vtable swap while other effects used evidence passing, effect-polymorphic code couldn't abstract uniformly. Handler forwarding (partially handling `Env.Read` while forwarding `Env.Write` to an outer handler) works naturally with evidence passing but requires fragile save/restore with vtable swap. The C runtime vtable should be an internal optimization detail of the default handler, not the dispatch mechanism. Counter-argument: Blink currently uses vtable swap for *all* effects, so this dissent is forward-looking — it applies when/if Blink migrates to evidence passing.
- **DevOps (A):** Handler blocks already emit save/restore in codegen for other effects. Adding env to the same pattern means handler diagnostics are consistent. `blink check` can verify handler completeness with the same logic used for IO/FS handlers. No new diagnostic surface needed.
- **AI/ML (A):** One handler mechanism for all effects = one pattern to learn. If env used evidence passing while IO used vtable swap, LLMs would need to learn two handler patterns. Consistency is the single strongest predictor of LLM generation accuracy.
